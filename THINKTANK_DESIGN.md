# 🚀 “智库 (ThinkTank)” 全栈开发指南 & 启动说明书

## 1. 项目定位
一个为技术团队设计的、文件原生的混合模式知识中心。它无缝融合了“在线协作编辑”与“Git仓库镜像”两种工作流，旨在提供极致的文档管理与阅读体验。

- **物理层 (Source of Truth)**：以磁盘文件系统为基石，所有 Markdown 文档和目录结构构成唯一的数据源。
- **逻辑层 (Control Plane)**：以“项目 (目录)”为核心单元，通过轻量化的 JSON 配置文件驱动元数据、分组、角色权限 (RBAC) 的管理，并为未来的 AI 向量搜索预留了扩展接口。
- **体验层 (User Experience)**：为“在线编辑”模式提供媲美本地 IDE 的实时编辑与文件管理体验；为“Git 镜像”模式提供高效、自动化的内容同步与纯粹的阅读体验，并通过统一的 UI 将两者有机结合。

---

## 2. 技术栈清单
- **全栈框架**：Next.js 14+ (App Router)
- **数据存储**：本地文件系统 (无关系型数据库依赖)
- **版本控制**：Git (通过 `simple-git` 库)
- **定时任务**：`node-cron` (用于 Git 自动同步)
- **编辑器**：Tiptap (Markdown)
- **UI 交互**：Tailwind CSS + shadcn/ui + sonner (通知) + react-arborist (文件树) + cmdk (命令菜单)
- **后端逻辑**：Node.js `fs` + `chokidar` (磁盘监听)
- **身份验证**：Auth.js (NextAuth) - 支持 GitHub/Google/Credentials
- **数据加密**：Node.js `crypto` (AES-256-CBC)
- **AI 引擎**：Vercel AI SDK (支持流式响应)

## 3. 功能模块设计

### 3.1 身份验证与权限体系
系统采用 **NextAuth.js** 构建了一套完整的身份验证体系，并实现了基于角色的访问控制 (RBAC)。

- **认证方式**：
  - **账号密码登录**：系统目前采用基于 `Credentials Provider` 的账号密码认证作为唯一的登录方式 (支持 Mock 数据：admin/123456, user1/123456)。
- **访问控制 (RBAC)**：
  - **游客 (Guest)**：
    - **权限**：只读访问。
    - **可访问区域**：
      - `/` (Preview Hub)：浏览所有项目卡片。
      - `/project/[id]/preview` (Project Preview)：阅读项目文档。
    - **限制**：不可创建项目、不可编辑/删除文件、无收藏夹/最近访问记录。
  - **管理员 (Admin)**：
    - **角色标识**：`role: 'admin'`
    - **权限**：拥有系统的完全控制权。
    - **能力**：
        - 创建、编辑、删除所有项目和分组。
        - 管理所有用户的权限。
        - 访问所有管理后台功能 (`/admin`)。
  - **普通用户 (User)**：
    - **角色标识**：无特定 `role` 字段或 `role: 'user'`。
    - **权限**：基于项目授权的读写访问。用户的权限被限制在 `accessibleProjects` 数组所指定的项目范围内。
    - **能力**：
        - 在被授权的项目中，可以编辑项目设置、创建/编辑/删除文档。
        - 访问 `/admin` 页面以查看和管理其有权访问的项目。
    - **数据结构**：用户的会话 (session) 中会包含一个 `accessibleProjects: string[]` 数组，其中列出了用户可以访问的所有项目 ID。API 会在执行操作前检查用户是否对目标项目有权限。

### 3.2 项目管理
- **项目制管理**：每个项目对应磁盘独立文件夹，支持两种创建模式：
  1. **Git 仓库镜像 (只读模式)**：
     - **用户配置**：
        - `Repository URL`: 必填，Git 仓库地址。
        - `Branch`: 必填，需要同步的分支名称。它决定了用户看到的“最新版”内容是基于哪个开发流。
        - `Root Path`: 选填，如果文档在仓库的子目录，则指定此路径。同步服务将只拉取此路径下的内容。
        - `Personal Access Token`: 选填，用于访问私有仓库。Token 会被 **加密存储**。
        - `Sync Interval`: 必填，自动同步的周期，单位为分钟。
     - **高效同步架构**：
        - **持久化克隆**：首次同步时，系统会在项目目录下的 `.temp/repo` 文件夹内完整克隆一次 Git 仓库。
        - **增量更新 (`pull`)**：后续同步时，不再重新克隆，而是进入 `.temp/repo` 目录执行 `git pull`，极大提升了同步效率。
        - **自动修复**：同步服务会自动检测本地仓库是否损坏或远程地址是否变更。如果发生变更，它会自动删除旧的 `.temp/repo` 并执行一次全新的克隆，保证数据源的正确性。
        - **内容复制**：拉取最新代码后，系统会将 `.temp/repo`（或指定的 `rootPath` 子目录）下的所有文件复制到项目的主目录中。项目主目录下的物理文件，永远是所选分支的最新版本。
        - **安全保留**：在复制内容前，系统会清空项目主目录，但会智能地保留 `.thinktank` (项目配置) 和 `.temp` (Git仓库) 两个关键文件夹，确保配置不丢失。
     - **动态调度**：当用户在设置页面修改了 `Sync Interval` 并保存后，系统会动态地停止旧的定时任务，并根据新的时间间隔创建一个新的定时任务，实现配置的即时生效。
     - **限制**：仅支持浏览，不允许在线编辑。

     - **版本管理 (Versioning)**：
        - **核心理念**：ThinkTank 作为远程仓库的“消费者”，只读取和展示远程仓库中已存在的 Git **标签 (Tags)**，而不在本地创建版本。
        - **同步增强**：后台同步任务在 `git pull` 之前，会先执行 `git fetch origin --tags`，确保所有远程标签都被同步到本地的 `.temp/repo` 仓库中。
        - **版本发现**：系统通过 `GET /api/projects/[id]/versions` API 提供版本发现能力。该 API 在后台执行 `git tag` 命令，并以语义化版本（SemVer）的顺序列出所有可用的版本标签。
        - **内容读取双轨制**：
            1.  **默认轨道 (最新版)**：当用户不选择任何特定版本时，API 直接从项目主目录读取物理文件。这些文件在同步时已被更新为所选分支的最新状态。
            2.  **虚拟轨道 (历史版本)**：当用户通过 UI 选择一个版本（如 `v1.0.0`）时，API 请求会附带 `?version=v1.0.0` 参数。后端此时将**忽略**项目主目录的物理文件，直接进入 `.temp/repo` 仓库，通过 `git show v1.0.0:path/to/file.md` 等命令，从 Git 数据库中“虚拟地”按需读取该版本快照的内容。
        - **前端体验**：
            - **版本切换器**：在项目页面，如果版本 API 返回了可用的标签列表，则自动出现一个版本切换下拉菜单。
            - **优雅降级**：如果远程仓库没有任何标签，版本切换器将不会被渲染，对用户完全无感。
            - **明确的视觉提示**：当用户浏览历史版本时，界面会给出清晰的提示（如“您正在查看归档版本 v1.0.0”），并禁用所有编辑入口。

  2. **在线编辑 (读写模式)**：
     - **用户配置**：项目名称、项目描述。
     - **系统行为**：在存储目录创建文件夹，全权管理文件生命周期。
     - **能力**：支持在线创建、编辑、删除文档及文件夹。

- **入口分离**：
  - `/admin`: **管理后台 (Admin Dashboard)** - 仅登录用户可见，提供项目和分组的管理功能。
  - `/`: **项目主页 (Project Hub)** - 游客/用户皆可访问，提供纯粹的阅读和导航体验。

### 3.3 项目分组管理
为了更好地组织和导航项目，系统引入了分组功能。

- **分组配置**：
  - 所有分组信息存储在 `docs/.thinktank/groups.json` 文件中。
  - 每个分组包含一个唯一的名称和其下的项目 ID 列表。
- **UI 展示**：
  - 在项目主页和管理后台，所有项目都将根据其所属分组进行归类展示，并以可折叠的标题栏分隔。
  - 未被分配到任何分组的项目，将统一显示在默认的“其他项目”类别下。
- **分组 CRUD**：
  - **创建 (Create)**：管理员可在后台的 "Group Management" 标签页中，通过 "Add Group" 按钮创建新分组。
  - **分配 (Update)**：
    1.  创建新项目时，可以通过一个可搜索的下拉框（Combobox）为项目指定一个现有分组，或直接输入新分组名进行创建。
    2.  在项目的设置页，管理员可以随时修改项目所属的分组。
  - **删除 (Delete)**：
    - 管理员可在后台删除分组。
    - **安全限制**：为防止误操作，只有当一个分组内不再包含任何项目时，其“删除”按钮才可被点击。否则按钮将置灰，强制要求管理员先将组内项目移出。

### 3.4 文档与文件管理
- **侧边栏文件树**：
  - **基础操作**：支持点击展开/折叠、选中高亮。
  - **拖拽排序**：支持文件/文件夹的拖拽移动（仅在线编辑模式），自动更新文件系统路径。
  - **右键菜单**：提供新建、重命名、删除、收藏、隐藏/显示等上下文操作（仅在线编辑模式）。
- **可见性控制**：支持文档隐藏。隐藏后在树目录中文字颜色变淡/斜体，且在预览入口不可见。
- **编辑器功能**：支持 Tiptap 源文件和预览文件的工作模式，自动大纲导航。
- **导出功能**：支持单篇 MD 导出及全项目 Zip 导出。

### 3.5 UI/UX 体验优化
- **项目设置页**：
  - **Token 安全输入**：Token 输入框默认显示为密码模式（`********`）。用户可点击旁边的“眼睛”图标，临时查看或隐藏真实 Token。
  - **连接测试**：在 Token 输入框的同一行，提供 "Test Connection" 按钮。点击后，系统会使用当前配置（包括处理 `********` 占位符）尝试连接 Git 仓库，并立即给出成功或失败的反馈。
- **管理后台**：
  - **卡片样式统一**：项目卡片的 UI 风格与预览主页保持一致，设计更简洁。删除按钮在鼠标悬停时才在右上角显示。
- **全局通知**：
  - 所有的操作反馈（如“保存成功”、“连接失败”等）都会以 Toast 的形式显示在屏幕顶部的正中央，位置统一且醒目。
- **侧边栏布局**：
  - **视觉分组**：在侧边栏顶部，当前登录用户的区域与下方的项目信息区域之间，增加了一条明确的 **分割线** 和额外的 **垂直间距**。
  - **目的**：通过清晰的视觉分组和留白，增强了界面的层次感，避免了用户区域和项目区域因布局过近而产生的主次不分的问题，提升了整体的可读性和美观度。

### 3.6 个性化功能
- **用户数据隔离**：用户的个性化数据（收藏夹、最近访问）存储于项目根目录下的 `.thinktank/users/[userId].json` 文件中。
- **首页 Dashboard**：
  - 展示项目描述（支持行内编辑）、README 预览。
  - **个性化板块**：
    - **Starred Documents**：显示用户收藏的文档（游客不可见）。
    - **Recently Updated**：显示用户最近访问/更新的文档（游客不可见）。

## 4. 逻辑删除与物理删除
为提升数据安全性和容错能力，系统对所有核心资源（项目、分组、文件/文件夹）的删除操作都设计了“逻辑删除”和“物理删除”两种模式。

- **核心机制**：
  - **逻辑删除 (默认)**：执行删除操作时，系统默认不会从磁盘上移除文件或从配置文件中删除条目。取而代之的是，在对应的配置项中添加一个 `isDeleted: true` 的标记。
    - **效果**：被逻辑删除的资源将在所有 UI 界面（如项目列表、文件树）中被隐藏，但其物理文件和数据依然存在，为数据恢复提供了可能。
  - **物理删除 (需二次确认)**：用户在执行删除操作时，可以通过勾选弹窗中的“永久删除 (Permanently delete)”复选框来触发物理删除。
    - **效果**：直接从磁盘上删除文件/文件夹，或从 JSON 配置文件中彻底移除相关条目。此操作不可逆。

- **实现细节**：
  - **确认弹窗**：所有删除操作都会弹出一个统一的确认对话框，其中包含触发物理删除的复选框。
  - **API 设计**：负责删除的 API 端点（如 `DELETE /api/projects/[id]`）会接受一个 `physical: boolean` 参数，用于区分执行哪种删除模式。
  - **数据读取**：所有的数据获取服务（如 `getProjects`, `getFileTree`）在返回数据前，都会自动过滤掉被标记为 `isDeleted: true` 的条目。
  - **恢复机制**：
    - **文件/文件夹**：当在同一路径下创建同名文件/文件夹时，系统会自动移除其 `isDeleted` 标记，实现“恢复”。
    - **项目/分组**：目前需要手动修改对应的 JSON 配置文件来恢复。

## 5. 数据结构约定
- **项目配置**：`docs/[projectId]/.thinktank/config.json`
  ```json
  {
    "description": "The official documentation for ThinkTank.",
    "mode": "git",
    "gitConfig": {
      "repoUrl": "https://gitlab.example.com/user/repo.git",
      "branch": "main",
      "rootPath": "docs/articles",
      "token": "encrypted_token_string_here", // AES-256-CBC 加密后的 Token
      "syncInterval": 5
    },
    "createdAt": "2024-01-01T00:00:00.000Z",
    "files": {
      "/getting-started.md": {
        "isDeleted": true // 文件级别的逻辑删除标记
      }
    },
    "isDeleted": false // 项目级别的逻辑删除标记
  }
  ```
- **分组配置**：`docs/.thinktank/groups.json` (全局唯一)
  ```json
  [
    {
      "name": "核心项目",
      "projects": ["thinktank-docs", "another-project"],
      "isDeleted": false
    },
    {
      "name": "实验性项目",
      "projects": [],
      "isDeleted": true
    }
  ]
  ```
- **用户配置**：`docs/.thinktank/users/[userId].json`
  ```json
  {
    "starred": ["/path/to/doc1.md"],
    "recent": ["/path/to/doc2.md"]
  }
  ```
- **文件元数据**：存储于 Markdown Frontmatter 中。
  ```markdown
  ---
  title: 文档标题
  isHidden: false
  sortOrder: 1
  createdAt: 2023-10-01
  ---
  ```

## 8. 项目优势与竞品分析

为了准确评估 ThinkTank 的核心价值，我们将其与 GitHub 上两类主流的开源文档项目进行对比：

1.  **静态站点生成器 (Static Site Generators - SSGs)**：如 **Docusaurus, Nextra, VitePress**。这类项目专注于“文档即代码”，将 Git 仓库中的 Markdown 渲染成静态网站。
2.  **在线协作 Wiki/编辑器平台**：如 **Outline, Wiki.js, HedgeDoc**。这类项目提供一个在线的、所见即所得的编辑环境，数据通常存储在数据库中。

### 8.1 竞品分析

| 特性 | Docusaurus / Nextra (SSGs) | Outline / Wiki.js (Wikis) | **ThinkTank (本项目)** |
| :--- | :--- | :--- | :--- |
| **核心模式** | Git 仓库 -> 静态网站 | 在线协作编辑器 | **混合模式 (Git + 在线编辑)** |
| **在线编辑** | ❌ 不支持 | ✅ **核心功能** | ✅ **核心功能 (在特定模式下)** |
| **Git 集成** | ✅ **核心功能 (只读)** | 🟡 有限支持 (通常是导出/备份) | ✅ **核心功能 (只读镜像 + 在线编辑)** |
| **数据源** | 文件系统 (Git 仓库) | 数据库 (PostgreSQL 等) | **文件系统 (无数据库)** |
| **部署复杂度** | 低 (静态文件) | 高 (需要应用 + 数据库) | **中 (仅需 Node.js 环境)** |
| **目标用户** | 开发者、开源项目 | 企业团队、非技术人员 | **技术团队、开发者、混合型团队** |

### 8.2 ThinkTank 的核心优势

通过对比，ThinkTank 的独特优势在于它并非“二选一”，而是“二者兼得”，精准地切入了市场空白。

1.  **独一无二的“混合模式” (The Hybrid Model)**
    - 这是项目 **最大、最核心的优势**。它没有强迫用户在“在线编辑”和“Git 工作流”之间做选择，而是将两者无缝地融合在同一个平台中。
    - **对于需要快速迭代、团队协作的内部知识库**，可以使用“在线编辑”模式，享受 Wiki 平台的便捷。
    - **对于需要严格版本控制、与代码一同发布的产品文档**，可以使用“Git 镜像”模式，享受 SSGs 的规范和自动化。
    - **结论**：其他项目要么是纯粹的 SSG，要么是纯粹的 Wiki。ThinkTank 是一个统一了两种模式的“知识中心”。

2.  **文件原生的轻量级架构 (File-Native & Lightweight)**
    - 与 Outline 或 Wiki.js 不同，ThinkTank **没有数据库依赖**。所有数据（文档、配置、用户权限）都以文件的形式存储在磁盘上。
    - **部署简单**：无需配置和维护数据库，极大降低了部署和运维的复杂度。
    - **数据透明**：所有数据都是人类可读的文件，可以直接查看、修改、备份和迁移。
    - **Git 友好**：这种架构天然与 Git 的工作方式相契合。

3.  **为开发者优化的集成体验 (Integrated Developer Experience)**
    - **统一的管理后台**：管理员可以在一个地方管理所有项目（无论是 Git 模式还是在线编辑模式）、分组和用户权限。
    - **智能的 Git 同步**：Git 镜像模式下的增量更新、自动修复和动态调度等设计，体现了对效率和稳定性的深度思考，超越了许多 Wiki 平台的简单“导出”功能。
